# 429 错误优化实现说明

## 已实现的优化功能

### 1. 429 错误临时禁用机制 ✅

**实现位置**：`monitor/manage.go`

**功能**：
- 当渠道返回 429 错误时，自动记录到缓存
- 在冷却期内（默认 5 分钟），该渠道会被临时阻止使用
- 自动清理过期的 429 记录

**关键函数**：
- `RecordChannel429(channelId int)`: 记录渠道 429 错误
- `IsChannel429Blocked(channelId int) bool`: 检查渠道是否被阻止
- `CleanExpired429Records()`: 清理过期记录

### 2. 指数退避重试策略 ✅

**实现位置**：`controller/relay.go`

**功能**：
- 重试时使用指数退避延迟
- 第 1 次重试等待 1 秒，第 2 次等待 2 秒，第 3 次等待 4 秒，以此类推
- 最大延迟时间可配置（默认 10 秒）

**关键函数**：
- `calculateBackoffDelay(attempt, baseDelay, maxDelay)`: 计算退避延迟

### 3. 智能渠道选择 ✅

**实现位置**：`controller/relay.go`

**功能**：
- 重试时自动跳过被 429 阻止的渠道
- 避免重复向已 429 的渠道发送请求

### 4. 配置项 ✅

**实现位置**：`common/config/config.go`

**新增配置项**：
- `CHANNEL_429_DISABLE_DURATION`: 429 错误临时禁用时间（秒），默认 300（5 分钟）
- `RETRY_BACKOFF_ENABLED`: 是否启用指数退避重试，默认 true
- `RETRY_BACKOFF_BASE`: 重试基础延迟（秒），默认 1
- `RETRY_BACKOFF_MAX`: 重试最大延迟（秒），默认 10
- `CHANNEL_429_AUTO_DISABLE`: 是否自动临时禁用返回 429 的渠道，默认 true

## 使用方法

### 环境变量配置

```bash
# 429 错误临时禁用时间（秒），默认 5 分钟
export CHANNEL_429_DISABLE_DURATION=300

# 是否启用指数退避重试，默认 true
export RETRY_BACKOFF_ENABLED=true

# 重试基础延迟（秒），默认 1
export RETRY_BACKOFF_BASE=1

# 重试最大延迟（秒），默认 10
export RETRY_BACKOFF_MAX=10

# 是否自动临时禁用返回 429 的渠道，默认 true
export CHANNEL_429_AUTO_DISABLE=true
```

### 代码逻辑流程

1. **请求处理**：
   - 正常请求 → 如果成功，返回结果
   - 如果失败且是 429 错误 → 记录到缓存，开始重试

2. **重试流程**：
   - 计算指数退避延迟并等待
   - 选择新渠道（跳过被 429 阻止的渠道）
   - 重试请求
   - 如果重试也遇到 429，继续记录并重试

3. **渠道管理**：
   - 429 错误自动记录到缓存
   - 冷却期内自动跳过该渠道
   - 冷却期后自动恢复使用

## 预期效果

实施这些优化后，预期可以：

1. **减少 429 错误率**：通过临时禁用和智能选择，减少向已 429 的渠道发送请求
2. **提高成功率**：指数退避给上游 API 恢复时间，提高重试成功率
3. **降低延迟**：减少无效重试，降低平均响应时间
4. **改善用户体验**：更快的响应和更高的成功率

## 注意事项

1. **内存使用**：429 缓存使用内存存储，但会自动清理过期记录，影响很小
2. **配置调优**：根据实际业务情况调整冷却时间和退避参数
3. **监控告警**：建议设置告警，当 429 错误率过高时及时通知
